## 1. Executive summary

Lupos is already a deep persona bot: server-side somatic/emotion engine (Plutchik + decay), ResponseVarietyService, embedding memory, a rich per-user dossier prompt, and a lean agentic `/agent` loop. The survey's strongest yield is therefore **not** "add a personality" ‚Äî it's three clusters:

1. **Make the invisible tamagotchi state legible.** Lupos computes mood/hunger/alcohol/energy every turn but the community never sees it. The highest-impact new capabilities (webhook per-message mood avatar, mood-reactive per-guild profile, consistent self-portrait selfies, communal `/feed` + caretaker leaderboard) surface that state cheaply.
2. **Make it smarter about WHEN it speaks.** Lupos is hard `@`-mention-gated with DMs off. A graduated free-will speak-gate, a value-scored interject gate, addressee recognition, and inbound-burst debounce turn a trigger-only tool into an ambient member ‚Äî all as cheap PRE-Prism gates.
3. **Turn data it already hoards into recurring rituals.** Full Mongo message archive + heatmap/wordcloud/deathroll data + Prism narration = persona-narrated Server Wrapped, nightly digest, `/catchmeup`, server-culture trivia, and `/whois` recall, at near-zero marginal infra.

Two platform-native persona wins (native polls keyed to mood; soundboard wolf-howls) and one genuine safety gap (crisis/self-harm break-character referral) round out the shortlist. Voice is the biggest single leap: **prism-service already ships a bidirectional Gemini Live `/ws/live` audio backend** and lupos already receives+decodes per-user Opus, so two-way VC conversation is wiring, not greenfield.

## 2. Method

**What was surveyed.** Companion/persona bots (Shapes.inc, Thea, Discord-Buddy, Nomi/Replika, Tatsugotchi/Disgotchi, Kodama, PluralKit/Tupperbox, Karuta, Dead Chat Reviver), self-hosted OSS LLM-Discord frameworks (llmcord, OpenClaw, hermes-agent, elizaOS, VoiceCordAI, openclaw-voice, Monitori), SillyTavern extensions (World Info, MemoryBooks, BetterSimTracker), academic MPC/turn-taking + emotion + injection work (Koala 2501.17258, Speak-or-Stay-Silent 2603.11409, Multi-Party survey 2505.18845, CPM 2607.07824, ModeratorLM 2606.13544, Smart Turn v3, Spotlighting, PromptGuard 2), and the Discord platform change-log itself (Enhanced Role Colors, Server Tags, Search Guild Messages, per-guild member profiles, Poll, Soundboard, Components V2, modal/context-menu expansion, AutoMod, DAVE, member-fetch rate limit, permission splits).

**How verified.** Each candidate was checked against primary sources (Discord dev docs/change-log, arXiv HTML, GitHub repos/stars/commit dates) for reality + recency, then grepped against the actual lupos-bot and prism-service trees to establish the real gap (`missing` vs `partial`) with file/line evidence, distinguish it from already-shipped or already-planned work (`docs/IMPROVEMENT_PLAN.md`, `agentic_integration_plan.md`, `TODO.md`), and produce a build sketch naming the exact service/job/command file, discord.js API, and any Prism-side hook. Several candidate claims were corrected where they overstated the gap or misattributed a source; the corrected `refinedClaim` supersedes the original pitch throughout. All impact/effort ratings below are the verifier's post-review values, not the scout's.

### 3.1 Media / visual expression ‚ú® (lead cluster ‚Äî makes the tamagotchi legible)

#### Webhook per-message mood avatar + name ‚ú® ‚Äî effort M, impact **High**
**Source:** [Discord Webhook API](https://docs.discord.com/developers/resources/webhook) ¬∑ [PluralKit guide](https://pluralkit.me/guide/)

**What + mechanism.** PluralKit/Tupperbox/Persona re-send messages through a channel webhook, overriding `username` + `avatar_url` per message via Discord's Execute Webhook API ‚Äî something a bot account cannot do for its OWN messages, and which sidesteps the ~2/hour `ClientUser.setAvatar/setUsername` cooldown (webhooks have their own generous ~30/60s-per-webhook, 5/5s-per-channel limits). GA API, no verified-bot/large-server gating; creating the webhook needs only MANAGE_WEBHOOKS, executing it needs no permission. Routing replies through a per-channel webhook lets each message carry a mood-portrait avatar + name suffix (e.g. `lupos (starving)` + hungry face) driven by the existing trait state. Caveat: webhooks work only in guild text channels/threads, not DMs.

**Lupos gap (missing).** No webhook send anywhere: grep for `createWebhook/fetchWebhooks/webhook.send` across src returns nothing; `webhookId` appears only as an incoming read in `transformers.ts:360/919`. All replies go through `message.reply()` (`DiscordService.ts:256,765,786,811`) and `DiscordUtilityService.sendMessageInChunks:278`. Somatic/persona state is `TraitRegistry.ts` (mood ‚àí10..10 with per-level emoji in `MoodConstants.ts`), exposed via `GET /bot/stats` (`GuildRoutes.ts:890-1012`) and consumed by Prism ‚Äî never surfaced on Discord messages. Self-loop already guarded: `DiscordService.ts:854` `isMessageFromBot = message.author.bot` (webhook messages have `author.bot=true`). `TODO.md` lists "Bot changes its avatar" but nothing webhook-per-message.

**Build sketch.** New `WebhookProxyService` (or method on `DiscordWrapper`) that lazily fetches-or-creates and caches one webhook per text channel: `(channel as TextChannel).fetchWebhooks()` ‚Üí reuse an owned one, else `.createWebhook({ name: 'lupos' })`. Send via `Webhook#send({ username, avatarURL, content, files, threadId })`. Wire into `DiscordUtilityService.sendMessageInChunks` by adding a `webhook` mode alongside the current `reply` mode (it already chunks text + attaches images/audio buffers), computing `username = lupos (${TraitRegistry.getMoodName()/status})` and `avatarURL` = a mood portrait. Portraits: pre-render ~11-21 mood faces (one per MOODS level) to MinIO once (`MediaArchivalService/MinioWrapper`) and map moodLevel‚ÜíURL; or generate on-demand via Prism image-gen and cache. Guard: cache webhook IDs and skip messages whose `webhookId` is ours; fall back to `message.reply()` in DMs and threads-without-webhook. **Tradeoffs to accept:** webhook replies render as a distinct pseudo-user with an APP tag, don't thread as native inline replies, and won't match `client.user.id`-keyed logic (`DiscordService.ts:850,1153`). Needs MANAGE_WEBHOOKS in the guild.

#### Mood-reactive per-server bot profile + custom-status presence ‚ú® ‚Äî effort M, impact Med
**Source:** [Discord change-log 2025-09-10](https://docs.discord.com/developers/change-log) ¬∑ [Oct 2025 dev newsletter](https://discord.com/developer-newsletter/october-2025)

**What + mechanism.** Since the change-log entry dated **2025-09-10**, a bot can set its OWN per-guild avatar, banner, and bio via Modify Current Member (`PATCH /guilds/{id}/members/@me`) ‚Äî the Guild resource params table now includes `avatar`, `banner` (base64 data-URI), `bio` (string), `nick`, with no verified-bot or server-size gating. discord.js 14.26.4 (installed) exposes it natively as `guild.members.editMe({avatar, banner, bio, nick})`. The custom-status half is NOT new ‚Äî Lupos already drives `setActivity(ActivityType.Custom)` dynamically. The new mechanism is the per-guild MEMBER profile (avatar/banner/bio) wired to somatic state: a hungover avatar / "3 days without food" bio / "feeling: hungry" status that the tamagotchi "wears" in the member list with no message sent.

**Lupos gap (partial).** Custom status already dynamic (`DiscordUtilityService.ts:655-657` `setUserActivity`, driven at `DiscordService.ts:131/253/549`). Global identity patch exists but wrong scope: `DiscordUtilityService.ts:689-705` `patchBanner` does `PATCH /users/@me` (GLOBAL, no avatar/bio). `setNickname` is the only per-member edit used. Per-guild avatar/banner/bio via `editMe` never called. Somatic driver (`StatService.ts`/`TraitRegistry.ts`) feeds only chat tone via `PromptBuilder.ts`, never presence/profile. `TODO.md:33/36` "Bot changes its avatar / server banner" mean global/guild-banner, not a per-member mood profile.

**Build sketch.** API native in installed discord.js 14.26.4: `GuildMemberManager.editMe({ avatar: Buffer|data-URI, banner, bio, nick, reason })`. (1) Add helper `editGuildProfile(client, guildId, {avatar?, banner?, bio?})` in `DiscordUtilityService.ts` next to `patchBanner`. (2) Small driver mirroring `ServerIconJob.ts` (already does per-guild icon rotation): read current somatic tier from `StatService/TraitRegistry`, map tiers ‚Üí bio string, custom status via `setUserActivity`, optional avatar/banner variant. (3) THROTTLE: avatar/banner/bio go through `editMe`, are rate-limited AND client-cached, so edit only on mood-TIER transitions, not every tick. (4) Assets: pre-render mood variants or Prism-gen + MinIO archive. No LLM/Prism call required (state is local). **Ship bio + mood custom status first (cheap, S), then avatar/banner variants** ‚Äî avatar is heavily client-cached so its payoff lags.

#### Consistent self-portrait "selfie" from a canonical reference sheet ‚ú® ‚Äî effort S, impact Med
**Source:** [Gemini image gen (Nano Banana)](https://ai.google.dev/gemini-api/docs/image-generation)

**What + mechanism.** Reference-conditioned image models (Google "Nano Banana" / Gemini 2.5 Flash Image, and **Nano Banana 2 / Gemini 3.1 Flash Image, Feb 2026**) let you pin a canonical character reference and condition every generation on it so a character keeps the SAME face/outfit/proportions ‚Äî the 2025-26 visual-consistency bar (also Ideogram Character, Flux Kontext, Leonardo Character Reference). Lupos already draws ITSELF generically (bribe/"visual retort" persona rules; a harness test literally says *"that's basically my self-portrait"*) but from a fresh text prompt every time, so its wolf-king looks different each render. Adding (1) a canonical Lupos reference sheet fed into `generate_image` on self-portraits and (2) a persona instruction to fold current somatic state (drunk/sick/hungry/exhausted) into the prompt upgrades generic self-gen into a recognizable recurring self that "takes a selfie" reflecting live mood.

**Lupos gap (partial).** Generic self-drawing exists: `prism-service/src/locales/en/personas/lupos.json` `toolPolicyCore` bribe/visual-retort rules ("draw a visual retort of yourself... greedily eating a tray of donuts"); harness test `messageArrayConstruction.test.ts:912`. NO canonical reference pinned: `LuposPersona.ts:228 avatar:"/lupos-agent-avatar.png"` is a dashboard UI avatar, never passed to `generate_image`. Misattributed building block: `ImageIntent.ts` ¬ß4 "self-reference detection" detects a USER drawing THEMSELVES (attaches the user's avatar), NOT the bot. Reference-conditioning path already wired: `PromptBuilder.buildAndGenerateReply` attaches reference `imageUrls` (lines 1565-1592) and `toolPolicyImagePrompt` already handles "Preserve persons/avatars exactly as they appear." The exact enabling model is already in the stack: `lupos-bot/images/rod.dev/*gemini-3.1-flash-image-preview.png`.

**Build sketch.** Create a canonical Lupos reference sheet (1-3 images: face/full-body/expression) once; store in `lupos-bot/images/` or MinIO. In `ImageIntent.ts` add a `botSelfPortraitIntent(text)` helper mirroring the existing SELF_REF regex tier ("draw yourself", "selfie", "show us you", bribe case) ‚Äî or simplest, always attach the canonical ref as a low-priority reference. In `PromptBuilder.buildAndGenerateReply`, when self-portrait intent fires, push the canonical image into `imageUrls` labeled "Lupos's canonical appearance (reference ‚Äî keep him consistent)". Prism hook: add a line to `lupos.json` `toolPolicyImagePrompt` telling Lupos on self-portraits to (1) stay faithful to the attached canonical reference and (2) fold his CURRENT somatic state into the prompt (the somatic block is already in the system prompt). **Add the caveman-locale parity string** (`caveman/personas/lupos.json`) or locale tests fail. Risk: reference drift on scene-heavy prompts ‚Äî start self-only, gate to explicit selfie/bribe triggers before making it always-on.
